# Chapter: IndexedSegment - The Building Block of Vector Search

## Introduction: What is an IndexedSegment?

Imagine you're organizing a massive library. You can't just throw books on shelves randomly and expect to find anything later. Instead, you create a catalog card for each book that includes not just the book's location, but also a unique fingerprint that captures what the book is "about" in a way that computers can compare.

An **IndexedSegment** is exactly that catalog card for our AI-powered search system. It's the fundamental building block that pairs a piece of text (like a paragraph from a document) with its mathematical fingerprint—called an **embedding**. This pairing is what makes lightning-fast semantic search possible.

Think of it this way:
- **TextSegment** = The actual book (your content)
- **Embedding** = The catalog card (mathematical representation)
- **IndexedSegment** = Both together, ready to be searched

## How It Works: The Two-Part Identity

An IndexedSegment has a simple but powerful job: keep a piece of text and its vector embedding together as a single unit. Here's why both parts matter:

### The TextSegment: Your Content
This is the actual text you want to search through—maybe a paragraph from a company policy document, a chunk of a support article, or a section from a technical manual. It contains:
- The raw text content
- Optional metadata (like the source document, page number, timestamps)

### The Embedding: The Mathematical Fingerprint
This is a vector (an array of numbers, typically 384 or 768 dimensions) that represents the *meaning* of the text. Similar concepts have similar embeddings, which is how semantic search works. The embedding is generated by machine learning models that understand language context.

When these two pieces come together as an IndexedSegment, you get the best of both worlds: searchable meaning AND readable content.

## Code Deep Dive

Let's examine the code for IndexedSegment:

```java
package com.techcorp.assistant.store;

import dev.langchain4j.data.embedding.Embedding;
import dev.langchain4j.data.segment.TextSegment;

/**
 * IndexedSegment pairs a text chunk with its vector embedding.
 *
 * This is the fundamental storage unit for our vector search system.
 * Think of it as a "searchable text container" where:
 * - segment = the actual text content we want to retrieve
 * - embedding = the mathematical representation used for similarity search
 */
public record IndexedSegment(TextSegment segment, Embedding embedding) {
}
```

### Why Use a Java Record?

You might notice this is a **Java record** rather than a traditional class. This is a brilliant choice for several reasons:

1. **Immutability**: Once created, an IndexedSegment cannot be modified. This is perfect for our use case—once text is embedded, it shouldn't change.

2. **Automatic Methods**: Records automatically provide:
   - Constructor: `new IndexedSegment(segment, embedding)`
   - Accessors: `indexedSegment.segment()` and `indexedSegment.embedding()`
   - `equals()` and `hashCode()`: Useful for comparing segments
   - `toString()`: Helpful for debugging

3. **Conciseness**: What would take 30+ lines in a traditional class is just one line with a record.

### Using IndexedSegment in Practice

Here's how you might create an IndexedSegment in your application:

```java
// Step 1: Start with your text
String documentText = "TechCorp offers 15 days of paid vacation annually.";

// Step 2: Create a TextSegment (usually done by DocumentChunker)
TextSegment segment = TextSegment.from(documentText);

// Step 3: Generate an embedding (usually done by EmbeddingService)
Embedding embedding = embeddingService.embed(documentText);

// Step 4: Combine them into an IndexedSegment
IndexedSegment indexed = new IndexedSegment(segment, embedding);

// Now this indexed segment can be stored and searched!
```

### Accessing the Data

Since it's a record, accessing the data is straightforward:

```java
// Get the text content
String text = indexed.segment().text();

// Get the embedding vector
float[] vector = indexed.embedding().vector();

// Both are immutable - you can read but not modify
```

## Relationships: How IndexedSegment Fits Into the System

IndexedSegment is the bridge between several key components in our vector search system:

### Creation Pipeline (Chapters to Read First)
1. **DocumentChunker** takes large documents and splits them into smaller TextSegment objects
2. **EmbeddingService** takes each TextSegment and generates an Embedding
3. These two pieces combine to create an **IndexedSegment**

### Storage and Retrieval (Related Chapters)
4. **VectorStoreService** stores collections of IndexedSegments in memory, making them searchable
5. During search, **VectorStoreService** compares query embeddings against stored IndexedSegments
6. **SearchMatch** wraps an IndexedSegment with a similarity score when returning search results

Here's the data flow:

```
Document (PDF/Text)
       ↓
[DocumentChunker splits it]
       ↓
TextSegment (chunk 1), TextSegment (chunk 2), ...
       ↓
[EmbeddingService embeds each]
       ↓
IndexedSegment, IndexedSegment, IndexedSegment, ...
       ↓
[VectorStoreService stores them]
       ↓
Ready for searching!
```

## Key Takeaways

- **IndexedSegment is a simple but essential data structure** that pairs text content with its vector embedding
- **It uses a Java record** for immutability, clarity, and automatic functionality
- **The TextSegment contains the readable text** that users will see in search results
- **The Embedding contains the mathematical representation** that makes semantic search possible
- **IndexedSegment is created during document ingestion** and stored in the VectorStoreService for later searching
- **It serves as the bridge** between content processing (chunking/embedding) and search operations
